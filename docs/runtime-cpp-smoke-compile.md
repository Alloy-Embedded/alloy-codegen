# Runtime C++ Smoke-Compile Gate

Added by the OpenSpec change `add-runtime-cpp-smoke-compile-ci`.

## What it is

For every admitted `(vendor, family, device)` triple, the gate
materialises the pipeline's emitted runtime headers to a tmp tree
and drives:

```
clang++ -std=c++20 -ffreestanding -nostdlib -fno-exceptions -fno-rtti -O0 -c smoke.cpp
```

over a synthesised `smoke.cpp` that includes every runtime
header for that device.  Catches regressions invisible to the
golden + string-presence assertions:

- Typed-enum specialisation arity mismatches.
- `concept ValidPinAssignment<...>` template constraints that
  reference symbols that don't exist.
- Missing `#include` headers between emitted `.hpp` files.
- ODR violations across emitted artefacts in the same TU.

## Activation

The gate is **opt-in** for local runs and **mandatory in CI**:

```bash
# Pytest CLI flag:
pytest --runtime-cpp-smoke

# Or via env var:
ALLOY_RUNTIME_CPP_SMOKE=1 pytest tests/test_runtime_cpp_smoke.py
```

Without the flag, `pytest -q` skips the gate so the default
local loop stays fast.  When `clang++` is not on `PATH`, every
parametrised case is **skipped** (not failed) — contributors
without a clang toolchain are not blocked from running unrelated
tests.

In CI the `bootstrap-family.yml` workflow installs `clang` and
runs `ALLOY_RUNTIME_CPP_SMOKE=1 pytest tests/test_runtime_cpp_smoke.py`
right after the standard pytest pass.

## What gets compiled

Per-device, the harness materialises every artifact under
`<vendor>/<family>/generated/runtime/**/*.hpp` plus
`<vendor>/<family>/generated/runtime/types.hpp` and synthesises a
`smoke.cpp` like:

```cpp
// Auto-generated by tools/runtime_cpp_smoke.py — do not edit.
#include "<vendor>/<family>/generated/runtime/types.hpp"
#include "<vendor>/<family>/generated/runtime/devices/<device>/peripheral_instances.hpp"
#include "<vendor>/<family>/generated/runtime/devices/<device>/pins.hpp"
…
extern "C" int alloy_codegen_runtime_cpp_smoke_main() { /* … */ return 0; }
```

The include order is fixed in
`tools/runtime_cpp_smoke.py:_HEADER_ORDER` to mirror the
emitter's topological dependencies (peripheral_instances first,
pin_validation last so its `static_assert`s see all already-
declared enums).

## Failure mode

A failed compile fails the pytest case with a message that
identifies:

- The full `(vendor, family, device)` label.
- The list of headers included in the synthesised `smoke.cpp`.
- The first 60 lines of the compiler stderr.

That is sufficient context for the reviewer to fix without
reproducing locally.

## What this gate does NOT do

- **Does not link.**  Linking would require per-device startup
  files and is the runtime-lite consumer-verification harness's
  job.
- **Does not run optimisation passes.**  Smoke compile is `-O0`
  — the goal is "does it parse and typecheck", not "is the
  codegen optimal".
- **Does not validate runtime behaviour.**  `static_assert`s
  inside emitted headers are exercised, but no instances of
  drivers are actually constructed.

## Adding a new family

`tools/runtime_cpp_smoke.py:_build_smoke_execution_context`
declares the source-override map per `(vendor, family)`.  Adding
a new family is a one-entry edit:

```python
("acme", "fam"): {"acme-source": str(fixtures_root / "acme-fixture")},
```

The harness picks up every device for that family from
`alloy_codegen.bootstrap.DEVICE_REGISTRY` and runs the smoke
compile automatically — no per-device wiring required.

## Performance budget

Currently 17 admitted devices compile in ~15 s wall clock on a
Linux runner (M-series ARM Mac is similar).  As the device set
grows, the gate parallelises trivially because each compile is
independent — pytest-xdist or a CI matrix shard absorbs the
linear scaling.
