# Design — add-publication-scale-features

## Context

Atualmente o publish workflow tem matrix estático de 9 famílias rodando em
sequência (`max-parallel: 1` por causa do checkout compartilhado do
`alloy-devices`).  Cada job leva ~50s–1m30s; total agregado por push é ~10min.

Dois problemas escalam mal:

1. Visibilidade do conjunto admitido: os 9 family.json + N device.json estão
   espalhados; usuários precisam ler todos pra mapear "MCUs e peripherals
   suportados".
2. Compute desperdiçado: um patch de família roda os 8 jobs irrelevantes.  Com
   1000 famílias, qualquer push triggers 1000 jobs em ~16h+ de CI.

Ambos têm a mesma raiz: o publish workflow trata todas as famílias homogeneamente,
sem aproveitar conhecimento sobre o que mudou.  Esta proposta adiciona dois
mecanismos que injetam esse conhecimento no fluxo:

- README é uma **projeção** completa do conjunto admitido (todas as famílias,
  sempre).
- Affected detection é uma **filtragem** do conjunto admitido (só as famílias
  cuja entrada de fato mudou).

São duas faces do mesmo eixo "publication ergonomics for scale".

## Decision 1 — README é gerado pelo publish stage como artifact

**Decisão**: o root `README.md` do `alloy-devices` é um `EmittedArtifact`
produzido pelo stage `publish` em `stages/publish.py`, no path `README.md`
(sem prefixo de vendor/família), com `artifact_kind="documentation"`.

**Razão**:
- Consistente com o resto da pipeline: tudo que vai pro `alloy-devices` passa
  pelo stage publish.  Nenhum mecanismo paralelo.
- `promote_staging_root` já trata top-level non-vendor children (commit `0153d33`).
  Sem código novo de promotion.
- Cada job paralelo escreve o mesmo conteúdo (input idêntico) → idempotente.
  `git status` no workflow não vê diff quando nada mudou e o commit é skipado.

**Consequência**: o README é regenerado em todo publish, mas só commitado quando
o conteúdo realmente mudou.  Manter manualmente é proibido — a primeira linha
do arquivo declara isso.

## Decision 2 — README schema e formato

**Decisão**: o README tem três seções fixas:

```markdown
# alloy-devices

> Auto-generated by alloy-codegen <revision>.  Do not edit manually — every
> publish run regenerates this file from `DEVICE_REGISTRY` and family patches.

## Admitted families

| Vendor | Family | ISA | Devices | Packages | Peripherals |
|--------|--------|-----|---------|----------|-------------|
| ... | ... | ... | ... | ... | ... |

## Coverage caveats

- **<vendor>/<family>**: <__readme_caveat from family.json>
- ...
```

A coluna **Peripherals** lista todos os admitted (sem truncar).  GitHub renderiza
tabelas markdown com horizontal scroll quando excede a largura — o trade-off é:
"informação completa + scroll" vence "tabela compacta + clique pra detalhe".

A seção **Coverage caveats** é colhida do campo opcional
`__source_notes.__readme_caveat` em cada `family.json`.  Famílias sem o campo
não aparecem na seção (mantém ela enxuta).

**Razão**:
- Schema fixo permite golden test trivial — drift detectado por byte equality.
- Caveats são **per-família** (não per-device) porque limitações geralmente são
  da família como um todo (ex.: ESP32 SVD parcial, RP2040 single-core).
- Auto-extração via `__readme_caveat` evita que o emitter precise saber a lista
  de quirks; quem documenta a limitação é o autor da família, no patch dela.

**Consequência**: novas famílias documentam suas próprias limitações.  Ninguém
edita `runtime_readme.py` quando admite uma nova família.

## Decision 3 — Affected detection é path-based, não AST-based

**Decisão**: a função `affected_families.compute_affected(diff_paths, registry)`
mapeia caminhos do git diff para o conjunto de `(vendor, family)` afetados via
**heurística de path matching pura**.  Sem leitura de AST, sem grafo de
dependências, sem instrumentação de import.

Mapping (em ordem de prioridade — primeiro match vence):

| Pattern | Famílias afetadas |
|---------|-------------------|
| `patches/<vendor>/<family>/**` | só `(vendor, family)` |
| `src/alloy_codegen/sources/<basename>.py` | famílias cujo `SOURCE_BUNDLES` contém o ID daquele source |
| `src/alloy_codegen/runtime_xtensa_startup.py` | famílias com `core` começando com `xtensa` |
| `src/alloy_codegen/runtime_riscv_startup.py` | famílias com `core` em `{rv32imc, rv32imac, riscv}` |
| `src/alloy_codegen/runtime_avr_startup.py` | famílias com `core` começando com `avr` |
| `src/alloy_codegen/runtime_startup.py` | famílias com `core` começando com `cortex-m` |
| Outros `src/alloy_codegen/**.py` | **todas** |
| `tests/**`, `openspec/**`, `*.md`, `.github/workflows/bootstrap-family.yml` | **nenhuma** |
| `.github/workflows/publish-alloy-devices.yml`, `pyproject.toml`, `uv.lock`, `.github/workflows/*.yml` exceto bootstrap-family | **todas** |
| (default fallback / unknown path) | **todas** |

**Razão**:
- AST/import-graph dá detecção mais precisa mas é complexo, frágil e exige
  manutenção quando refatoramos.  Path-based é "good enough": vendors são
  isolados em `sources/<vendor>.py`, arquiteturas em `runtime_<arch>_startup.py`,
  famílias em `patches/<v>/<f>/`.
- Conservador por default: qualquer caminho não-mapeado dispara o matrix completo.
  Falsos positivos (rebuilds desnecessários) são preferíveis a falsos negativos
  (publicar artefato stale).
- O ganho real vem do caso comum: "patch de uma única família" só roda 1 job.

**Consequência**: refatorações grandes (mover `sources/cmsis_svd.py` pra outro
pacote, por exemplo) podem invalidar o mapping e disparar matrix completo
silenciosamente.  Isso é **aceitável** — é o lado conservador da heurística.
Mitigação: o `force_all` workflow_dispatch input cobre o caso oposto (forçar
matrix completo quando alguém suspeita).

## Decision 4 — `affected-families` CLI é primitivo, não plugin

**Decisão**: novo subcomando `python -m alloy_codegen.cli affected-families
--since <ref>` lendo o git diff via `git diff --name-only <ref>...HEAD` e
imprimindo o resultado em JSON ou linhas plain-text.

```bash
# JSON shape (default --json):
{
  "since": "abc1234",
  "head": "def5678",
  "all_families": false,
  "families": [
    {"vendor": "espressif", "family": "esp32"},
    {"vendor": "st",        "family": "stm32g0"}
  ]
}

# Plain (linhas):
espressif/esp32
st/stm32g0
```

**Razão**:
- O workflow consome JSON via `fromJson()` — nada de parse complicado em bash.
- `all_families: true` é um flag que o workflow checka antes de tentar parsear
  matrix items (atalho).
- Plain text mode é util pra debug local (`gh run view --log` e copiar/cola).

**Consequência**: o CLI fica primitivo.  Quem quiser previewer o README localmente
ainda passa pelo publish stage (ou roda os emitters direto via Python).

## Decision 5 — `force_all` é workflow_dispatch input, não env var

**Decisão**: o input `force_all: boolean (default: false)` aparece no
`workflow_dispatch` do publish workflow.  Quando `true`, o `detect-affected`
job retorna o matrix completo bypassando o git diff.

```yaml
on:
  workflow_run:
    workflows: ["Bootstrap Family CI"]
    types: [completed]
    branches: [main]
  workflow_dispatch:
    inputs:
      force_all:
        description: "Bypass affected-families detection; run full matrix"
        type: boolean
        default: false
```

**Razão**:
- `workflow_dispatch` já é o canal pra runs manuais (já tem o input
  `push_changes`).  Adicionar `force_all` reusa o canal sem inventar mecanismo.
- Restrito a manual: o trigger `workflow_run` (encadeado do Bootstrap CI) sempre
  usa detection — não há jeito de "esquecer" o `force_all` ligado.
- Não polui env vars.

**Consequência**: cache invalidation / security re-publish exige push manual via
GitHub UI ou `gh workflow run`.  É operação rara e visível.

## Decision 6 — Affected-families spec mora em `vendor-admission`

**Decisão**: o requirement do affected-families CLI + workflow vai pra
`vendor-admission` (não `artifact-contract`).

**Razão**:
- `vendor-admission` já trata "qual é o conjunto admitido e como o CI o exercita"
  — adicionar "como o CI decide quais admitidos rodar agora" é evolução natural.
- `artifact-contract` é sobre **o que cada artifact contém**, não sobre **quais
  artifacts são gerados quando**.  README cabe em artifact-contract; affected
  detection não.

**Consequência**: dois delta files (artifact-contract + vendor-admission), cada
um focado.

## Open Questions

Nenhuma — todas as decisões respondem aos inputs do usuário (force_all sim,
path mapping refinado sim, README do alloy-codegen atualizado sim).
