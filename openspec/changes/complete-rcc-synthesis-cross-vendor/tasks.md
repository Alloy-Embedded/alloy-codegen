# Tasks — complete-rcc-synthesis-cross-vendor

## Phase 1: ESP32 family (DPORT / SYSTEM / PCR)

- [ ] 1.1 Audit every admitted ESP32 chip (esp32, esp32c3,
      esp32s3) and produce a `(chip, peripheral, register, field)`
      truth table by walking the YAML's
      `templates["dport"|"system"|"pcr"].fields` and matching
      against `device.peripherals[].id`.  Land it under
      `alloy_codegen/ir/vendor_tables/espressif_clock_gates.py`
      so each row has `(chip_id, peri_id, en_path, rst_path)`.
- [ ] 1.2 Add `_synthesise_esp32_rcc(device)` to
      `alloy_codegen/ir/synthesised/builder.py`.  Dispatches on
      `device.identity.family` to the right chip's gate table.
- [ ] 1.3 Set `extra["bus"] = "DPORT"|"SYSTEM"|"PCR"` based on
      which template the gate field came from so the alloy HAL
      can dispatch.
- [ ] 1.4 Trait emitter: ensure the new `kBus` values pass through
      the existing `kBus = "<bus>"` constexpr line — no emitter
      changes required.
- [ ] 1.5 Regression test:
      `tests/test_rcc_synthesis_per_vendor.py::test_esp32_family`
      asserts each admitted ESP32 chip's `syn.per_rcc_map` size
      ≥ 30 (was 4–6) and that at least one peripheral has both
      `en` and `rst` paths.

## Phase 2: NXP iMXRT CCGR table

- [ ] 2.1 Hand-curate `alloy_codegen/ir/vendor_tables/nxp_imxrt_ccgr.py`
      with the iMXRT 1060 PID→(ccgr_idx, cg_idx) table from
      RM0090 §14 (Table 14-5).  Source comment in the module
      docstring lists the RM page each entry came from.
- [ ] 2.2 `_synthesise_imxrt_ccgr(device)` reads the table and
      populates `per_en[per_id] = "ccm.ccgr<N>.cg<M>"` for every
      peripheral that appears in the table AND in
      `device.peripherals`.  Skips peripherals NOT in the table
      (cross-checks against existing inline `rcc:` to avoid
      double-population).
- [ ] 2.3 Verify on mimxrt1062: `syn.per_rcc_map` size jumps from
      30 to ≥ 80 (covers ~80 of the 88 previously-empty
      peripherals; the rest are CPU-private resources without
      gates).
- [ ] 2.4 Regression test:
      `tests/test_rcc_synthesis_per_vendor.py::test_imxrt_ccgr_table`
      spot-checks a half-dozen well-known mappings (e.g.
      `lpuart1` ⇒ `ccm.ccgr5.cg12`, `gpio1` ⇒ `ccm.ccgr1.cg13`)
      and asserts that the inline `rcc:` blocks already declared
      on the YAML are NOT shadowed by the synthesised entry.

## Phase 4: AVR-Dx + NRF52 "no per-peripheral gate" marker

- [ ] 4.1 Add `gate_model: str | None` field to
      `alloy_codegen/ir/v2_1/peripherals.py` `PeripheralRcc`.
      Stays optional; default `None` means "synthesised by the
      vendor branch above and gated as-emitted".
- [ ] 4.2 Add a final pass in `_build_rcc_lookup` that, when
      `device.identity.family in {"avr-da", "nrf52"}` AND
      `per_rcc_map[per_id]` is empty, sets `gate_model =
      "always-on"` for every peripheral.  Result: the per_rcc_map
      goes from empty to "every peri carries a marker entry".
- [ ] 4.3 Define `enum class GateModel` in the alloy boundary
      (header generated by the codegen, not hand-written) with
      enumerators `always_on`, `per_peri_en`, `per_peri_en_rst`,
      `index_based`, `per_peri_pcr`.
- [ ] 4.4 Trait emitter: emit
      `static constexpr GateModel kGateModel = …;` on every
      peripheral instance.  Defaults to `per_peri_en_rst` when
      both `kRccEnable` and `kRccReset` are populated, falls
      back to `per_peri_en` for resets-only or enables-only,
      `index_based` when the synth came from
      `nxp_imxrt_ccgr.py`, `always_on` for AVR-Dx / NRF52,
      `per_peri_pcr` for ESP32-C3/S3 PCR-self-contained.
- [ ] 4.5 Regression test:
      `tests/test_rcc_synthesis_per_vendor.py::test_no_gate_model_marker`
      asserts every avr-da / nrf52 peripheral gets
      `kGateModel = GateModel::always_on` and no `kRccEnable`.
